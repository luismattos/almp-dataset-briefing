Executive Summary

O projeto: Transformar ~100 questionarios Word (.docx) preenchidos em entrevistas sobre programas de mercado de trabalho (ALMPs) num dataset Excel compativel com STATA.

O desafio central: Os questionarios sao semi-estruturados com 573 variaveis, 478 checkboxes, e formatos mistos de resposta. A complexidade real depende de como os ~100 entrevistadores preencheram — e ainda nao temos amostras reais.

Abordagens de Automacao

6 abordagens analisadas. Cada uma com trade-offs diferentes de esforco, risco, e qualidade.

RECOMENDADA: Python + IA Hibrido

Como funciona: Python (python-docx) extrai a estrutura das tabelas de forma deterministica. IA (Claude) interpreta o conteudo das celulas de resposta — especialmente campos ambiguos, multi-selects, e condicionais.

Esforco7-12 dias (script) + QA
Risco tecnicoMedio-Baixo
QualidadeAlta (melhor dos dois mundos)
Custo API$15-40 (tokens Claude)
Por que:Python garante extracao deterministica da estrutura. IA resolve o problema mais dificil: interpretar respostas ambiguas. Reduz custo de tokens (envia so tabela, nao doc inteiro).

Opcao B: IA-Assisted Puro (Claude/GPT le cada .docx)

Cada .docx convertido para texto e enviado a um LLM com schema JSON. O LLM retorna dados estruturados.

Esforco8-12 dias
RiscoMedio (alucinacao possivel)
QualidadeMedia-Alta (~95% consistencia)
Custo API$30-80
Trade-off:Rapido de prototipar, lida com variabilidade, mas pode inventar dados em campos ambiguos.

Opcao C: Full Python (python-docx + pandas)

Script Python puro que parseia tabelas, detecta checkboxes, mapeia variaveis.

Esforco12-18 dias
RiscoMedio-Alto (edge cases)
QualidadeAlta (deterministico)
Custo$0
Trade-off:Reprodutivel e sem custo, mas requer Python intermediario e muito tempo em edge cases.

Opcao D: Qualtrics Hybrid (manual via formulario)

Recriar questionario no Qualtrics. Operadores transcrevem respostas dos .docx manualmente.

Esforco15-25 dias
RiscoBaixo
QualidadeAlta (validacao built-in)
Custo$0-500 (licenca Qualtrics)
Trade-off:Zero codigo, mas lento e tedioso. Fadiga do operador = erros.

Opcao E: n8n Workflow Automation

Workflow visual: le .docx → converte texto → LLM node → JSON → Excel.

Esforco10-15 dias
RiscoMedio-Alto
QualidadeMedia-Alta
Trade-off:Visual e reusavel, mas menos flexivel para debugging de edge cases.

Opcao F: Entrada Manual em Excel

Template Excel com ~573 colunas. Abrir cada .docx lado a lado e digitar.

Esforco40-60 dias (1 pessoa)
RiscoAlto (erro humano)
QualidadeVariavel (fadiga)
Trade-off:Ultimo recurso. Inviavel para 100 questionarios.

Achados Tecnicos (Analise XML)

Analise direta da estrutura interna dos arquivos .docx.

Boas Noticias

  • Checkboxes sao Form Fields reais do Word (w:ffData/w:checkBox) — nao Unicode, nao imagens, nao Wingdings. Isso e otimo: o estado (checked/unchecked) e legivel diretamente no XML.
  • Estrutura de tabela unica e consistente — 1 tabela, 109 linhas, 3 colunas (label, pergunta, resposta).
  • Codebook STATA-compatible — 573 variaveis, ZERO problemas de compatibilidade (nenhum nome > 32 chars, nenhum espaco, nenhum char especial, nenhuma duplicata).

Alertas Criticos

  • Questionarios fake NAO usam os checkboxes — O exemplo preenchido do Kenya tem 409 checkboxes mas TODOS desmarcados. As respostas foram digitadas como texto nas celulas, nao clicando nos checkboxes. Isso significa que os fakes nao sao confiaveis como referencia de formato.
  • NAO sabemos como os questionarios REAIS foram preenchidos — Respondentes reais podem ter: (a) clicado nos checkboxes, (b) digitado texto, (c) deletado opcoes nao-selecionadas, (d) feito qualquer combinacao. Isso e o maior risco desconhecido.
  • 478 checkboxes no template vs. 409 no fake — 69 checkboxes "sumiram" no preenchimento. Respondentes podem ter deletado linhas ou modificado a estrutura.

Pre-Mortem: Analise de Riscos

"Estamos em maio 2026 e o projeto fracassou. O que deu errado?"

1. Questionarios reais nao seguem o template

Respondentes modificaram formatacao, tabelas quebradas, celulas fundidas, texto fora das caixas, formatos inesperados.

Probabilidade: ALTA Impacto: ALTO

Mitigacao: Pedir 3-5 questionarios reais ANTES de assinar. Se o cliente recusar, nao aceitar.

2. Codebook AI-generated esta errado

Variaveis mal nomeadas, codificacoes inconsistentes, secoes faltando. Cliente rejeita primeira entrega.

Probabilidade: ALTA Impacto: ALTO

Mitigacao: Exigir codebook oficial da ILO ou incluir criacao/validacao no escopo (e no preco).

3. Escopo flutuante — cliente espera mais que "um Excel"

Expectativa de limpeza de dados, harmonizacao, .dta STATA com labels, analises exploratorias, documentacao completa.

Probabilidade: ALTA Impacto: ALTO

Mitigacao: Lista exata de deliverables por escrito ANTES de assinar. Quantas rodadas de revisao.

4. Extracao por IA nao e confiavel o suficiente

Claude alucina valores, tabelas complexas parseadas incorretamente, taxa de erro acumulada nos 100 questionarios.

Probabilidade: MEDIA-ALTA Impacto: ALTO

Mitigacao: Pipeline com validacao humana obrigatoria. Nao prometer 100% automacao.

5. Subestimar tempo para edge cases

"Cauda longa" de problemas: respostas ambiguas, formatos numericos variados, inconsistencias internas. 40.000-50.000 data points para validar.

Probabilidade: ALTA Impacto: MEDIO-ALTO

Mitigacao: Buffer de 50% sobre estimativas. Entrega em fases com go/no-go.

6. Problemas de qualidade dos dados culpam o executor

Dados de origem ruins (respostas incompletas, contraditorias). Sem criterio claro sobre tratamento.

Probabilidade: MEDIA Impacto: ALTO

Mitigacao: Exigir protocolo de decisoes + log de todas decisoes tomadas.

7. Dependencia de Python sem experiencia

Bugs sutis (encoding, Unicode, tipos) consomem tempo desproporcional.

Probabilidade: MEDIA Impacto: MEDIO

Mitigacao: Abordagem hibrida. Ou 1 dia de dev Python para montar pipeline base.

8. Prazo irrealista para uma pessoa

100 questionarios x 450+ variaveis. Tempo efetivo pode ser 3-4 semanas com outras obrigacoes.

Probabilidade: MEDIA Impacto: ALTO

Mitigacao: Calculo realista de dias efetivos disponiveis. Negociar prazo ou escopo.

9. Formatos .docx especificos

Alguns podem ser .doc antigo, protegidos, com Track Changes, versoes diferentes do template, ou criados em LibreOffice.

Probabilidade: MEDIA Impacto: MEDIO-ALTO

Mitigacao: Confirmar que TODOS sao .docx e mesma versao do template.

10. Contrato nao cobre iteracoes

Revisoes sem fim. Cada rodada quase tao trabalhosa quanto a primeira entrega.

Probabilidade: MEDIA Impacto: MEDIO

Mitigacao: Max 2 rodadas de revisao no contrato. Revisoes estruturais = aditivo.

Deal-Breakers & Regras de Decisao

NAO aceitar se:

  • O cliente recusa enviar questionarios reais antes da assinatura
  • Nao existe codebook oficial e esperam que voce crie E valide (duplica/triplica escopo)
  • Prazo inferior a 6 semanas para uma pessoa
  • Questionarios em multiplos idiomas sem traducao prevista
  • Parte dos questionarios sao escaneados/PDFs de papel (OCR = projeto diferente)

Cautela extra se:

  • Respostas vagas sobre escopo e deliverables
  • "A gente ve isso durante o projeto" como resposta a questoes criticas
  • Nao ha pessoa tecnica de contato na ILO para duvidas durante execucao
  • Os documentos preparatorios sao todos AI-generated (ninguem da ILO validou?)
  • Ja tentaram antes e falharam (por que desta vez seria diferente?)

Sinal verde se:

  • Fornecem 5+ questionarios reais e a maioria segue o template
  • Codebook oficial existe e esta alinhado com o template
  • Prazo de 8+ semanas com entrega em fases
  • Todos os questionarios sao .docx em ingles
  • Ponto de contato tecnico dedicado para duvidas

Entrega Minima Viavel (se tudo der errado)

Se o projeto descarrilar, o que preserva sua reputacao:

  1. Pipeline funcional para o "happy path" — funciona nos 40-60% limpos
  2. Dataset parcial + log de problemas — documentacao de por que os demais falharam
  3. Template validado — estrutura correta mesmo que dados incompletos
  4. Manual de procedimento — para outra pessoa completar manualmente
  5. Proposta de Fase 2 — escopo do que falta, quanto tempo, quanto custa

Contexto Estrategico ILO

  • A ILO ja tem o ALMP Compendium para America Latina — este projeto e provavelmente uma expansao global
  • A ILO investe em automacao de dados (ILOSTAT) — valorizam pipelines reprodutiveis, nao processos one-off
  • STATA e a ferramenta padrao institucional da ILO para processamento de microdados
  • O questionario anual do ILOSTAT ja usa Excel padronizado — eles conhecem as dificuldades de padronizacao
